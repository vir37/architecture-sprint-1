Спринт 1, Задание 1
1.  В качестве инструмента для объединения микрофронтендов в одно приложение был выбран Webpack Module Federation.
    Почему? Просто потому, что вполне подходит для решения данной задачи (а также имеет хорошие примеры на GitHub)
    Я далек от фронтэнда и JS, разбираться в тонкостях реализации обоих инструментов заняло бы много времени, 
    а глобально эти фреймворки похожи. В общем удачно попавшиеся примеры решили вопрос в пользу WMF.
2.  https://github.com/vir37/architecture-sprint-1.git
    Структура кода.
    Монолитный фронтенд был разбит на 3 микрофорнтенда согласно их бизнес-функциональности (т.е. использовался подход DDD).
    Все проекты фикрофронтендов расположены в папке frontend\microfrontend
    а) auth-microfrontend - приложение, отвечающее за аутентификацию и регистрацию пользователей. Выделен в отдельный микрофронтенд
        для удобства дальнейшего развития (добавления новых механизмов аутентификации), масштабирования и повышения отказоусточивости.
        Структура:
        src\blocks - папка с наборами стилей
        src\components - папка с основными компонентами фреймворка
        src\components\AuthApp.js - главная компонента
        src\components\Header.js - компонента заголовка контента
        src\components\InfoTooltip.js -  компонента информационных сообщений
        src\components\Login.js - компонента, реализующая форму аутентификации пользователя
        src\components\Register.js - компонента, реализующая форму регистрации пользователя
        src\images - папка с картинками, используемыми в проекта
        src\utils - папка со вспомогательными утилитами
        src\utils\api.js    - набор методов API бэкенда для получения информации о пользователе  
        src\utils\auth.js   - набор методов API бэкенда для нужд аутентификации и регистрации
    
    b) card-microfrontend - приложение, отвечающее за функциональность работы с фотографиями. 
        Структура:
        src\blocks - папка с наборами стилей
        src\components - папка с основными компонентами фреймворка
        src\components\CardsApp.js - главная компонента
        src\components\Card.js - компонента карточки фотографии
        src\components\AddPlacePopup.js - компонента формы добавления новой карточки
        src\components\ImagePopup.js - компонента выбраной карточки
        src\components\PopupWithForm.js - общая компонента popup-форм
        src\images - папка с картинками, используемыми в проекта
        src\utils - папка со вспомогательными утилитами
        src\utils\api.js    - набор методов API бэкенда для работы с фотографими  

    c) profile-microfrontend - приложение, отвечающее за функциональность работы с профилем пользователя. Функциональность может развиваться самостоятельно,
        т.к. зависимость с другими модулями минимальна.
        Структура:
        src\blocks - папка с наборами стилей
        src\components - папка с основными компонентами фреймворка
        src\components\ProfileApp.js - главная компонента
        src\components\EditAvatarPopup.js - компонента формы редактирования аватара пользователя
        src\components\EditProfilePopup.js - компонента формы редактирования профиля пользователя
        src\components\PopupWithForm.js - общая компонента popup-форм
        src\images - папка с картинками, используемыми в проекта
        src\utils - папка со вспомогательными утилитами
        src\utils\api.js    - набор методов API бэкенда для работы с фотографими 

    frontend\shared-library - библиотека с разделяемым между микрофронтами контекстом. 
3.  В этой части ничего сделать не удалось, т.к. не силен в JS.

Монолит был разбит на следующие сервисы согласно их предметной области и функциональному назначению.
1) "Регистрация пользователя". Сервис предназначен для обработки заявок на регистрацию новых пользователей. Может содержать например набор правил валидации. 
Регистрирует пользователя на площадке, создает профиль.
2) "Аутентификация". Сервис предназначен для проведения аутентификации пользователя, проверки существования авторизующегося пользователя, выдачи токена JWT. Легко реализовать различные методы аутентификации. 
3) "Работа с заявками". Сервис предназначен для обработки разных заявок пользователя (регистрация, тех.поддержка, участие в аукционе). Легко добавлять новые виды заявок и логику их обработки.
4) "Профиль пользователя". Сервис предназначен для работы с профилем пользователя. Предоставляет API как фронтенду, так и бэкенд-сервисам. Легко добавлять новую функциональность.
5) "Торговая площадка". Сервис предназначен для управления торговыми площадками: создания, удаления, редактирования, предоставления информации другим сервисам.
6) "Уведомления". Сервис предназначен для реализации различных методов уведомления пользователей. Легко добавлять новые методы.
7) "Аукцион". Сервис предназначен для управления аукционами: создание, редактирования, удаления, управления сделками, регистрации пользователей в аукционах. 
8) "Услуги пользователя". Сервис предназначен для управления услугами пользователя: добавление, удаление, изменение, взаимодействие с каталогом услуг.
9) "Товары пользователя". Сервис предназначен для управления товарами пользователя: добавление, удаление, изменение, взаимодействие с каталогом товаров.
10) "Каталог товаров". Сервис предназначен для управления каталогом услуг.
11) "Каталог услуг". Сервис предназначен для управления каталогом товаров.
12) "Управление заказом". Сервис предназначен для управления заказами пользователя: добавления, редактирования, подтверждения, получения статусов.
13) "Платежи". Сервис предназначен для обработки платежей. Реализует интеграцию с внешними платежными сервисами, отслеживает состояние оплат. Легко добавлять новые внешние платежные сервисы.
14) "Аналитика и отчетность". Сервис предназначен для реализации различных видов отчетов, получени и агрегирования информации с различных сервисов: каталоги, модуль заказов.
